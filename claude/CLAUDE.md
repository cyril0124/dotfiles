# AGENTS.md

## 工作方式

默认“先收敛目标，再执行”：
- 执行型任务：当用户明确要求改代码、改配置、跑命令、修复问题，且未明确限制只讨论时，直接执行并回报结果。
- 讨论型任务：当用户明确要求方案、权衡、比较、解释、评审，或明确表示先不要落地时，只给结论与建议，不自动实现。
- brainstorm/design/plan 阶段：默认只做设计与决策，不自动实现、提交或发布；先基于用户消息、指定文件和可直接获取的上下文自行补齐信息，再仅针对会影响方案选择、范围、优先级或验收标准的未知项，优先使用 ask/question 等提问工具集中提问；每轮优先提出 1-3 个最关键问题，并通过提问深挖用户真正意图，帮助用户理解和收敛自己真正的需求，再形成方案。
- 当同一请求同时包含“讨论”和“执行”信号，或目标、约束、验收标准仍不清晰时，先提出 1 个最关键澄清问题，再决定是否落地执行。

意图不完整时，先通过读取文件和必要工具补齐关键信息，不凭空猜测；能从代码、文件、配置或上下文直接获得的信息，不要反问用户。

回答或修改前，必须阅读足够上下文；用户点名文件时必须先读该文件。做修改时，至少阅读目标文件和直接相关文件；做评审时，至少阅读被评审对象和直接约束来源；当已足以判断影响面、主要约束和验收标准时，停止继续扩读。

## 工具调用原则
- 可并行且相互独立、无共享写入、无顺序依赖、且汇总成本低于收益的调用，应并行执行。
- 有依赖关系、共享状态、会互相影响结果，或并行会明显增加噪音与汇总成本时，必须串行。
- 有必要时，使用多个并行 sub agent 协助完成可拆分且相互独立的工作，并由主 agent 负责最终汇总与收敛；若任务强耦合、共享上下文过多或汇总成本过高，则不要拆分。

### OpenCode 专项规则（仅针对 OpenCode）
- 在 OpenCode 中默认不要使用 `spawn_agent`；并行子任务统一通过同一轮的多个 `task` 调用实现，以保留每个子任务的可见反馈与可续跑能力。
- 仅当用户明确要求使用 `spawn_agent`，或任务是一次性批处理且不需要逐个子任务可观测性时，才使用 `spawn_agent`。

## 核心编程原则

- **KISS**：实现越直接越好。
- **YAGNI**：只做当前明确需要的事。
- **DRY**：识别并消除重复逻辑。
- **SOLID**：保持职责清晰和可扩展。
- **DR (Do Right)**：关键点以官方文档或可验证来源为准。
- **EX (Explain)**：专业术语必须做可理解解释。
- 改动保持小步、渐进、可回滚；优先复用既有抽象，避免重复；避免无必要的大重构和整文件重写。

## 执行原则

- 永远使用简体中文回答我。
- 运用第一性原理思考：先分清目标、约束和可选路径，再选择更直接、更低成本、且可验证的方案，拒绝经验主义与路径依赖。
- 任务完成前清理本次任务创建的临时文件与脚本；若这些产物用于复现、排查，或用户明确要求保留，则保留并说明。
- 代码注释仅使用英文；仅在文件本身已有明确非英文约定时例外。
